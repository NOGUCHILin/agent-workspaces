# 査定スキル（データ整理）

iPhone買取の査定時にKintoneレコードを更新するスキル。付属品設定から本査定完了メッセージの生成まで、査定に必要な全ての情報を入力します。

## 必要な入力情報

1. **レコード番号**（6桁）
2. **3utools画像のファイルパス**
3. **付属品の有無**
   - 箱: あり/なし
   - 充電ケーブル: あり/なし
   - 説明書・シムピン: あり/なし
   - ※充電アダプタ・イヤホンは機種から自動判定
4. **外装の状態**（表面・背面・側面）
   - まったく劣化がない / ちいさな傷などがある / 使用感がある / 目立つ傷などがある / 割れやゆがみがある
5. **不良の内容**（症状対応表A列から選択: BTサービス、OC曇り薄い、光抜け等）
6. **SIMロック状態**（実測値）
7. **バッテリー最大容量**（実測値、3utoolsの値と異なる場合がある）
8. **査定写真**（端末の傷や不良を撮影した画像4枚程度）
9. **販売経路情報**
   - 出品媒体（バックマーケット等）
   - BMコンディション（A/B/Cグレード）
   - 想定販売価格（Google Sheetsから取得、または手動入力）

## 査定者が渡す情報（自動化に必要な9項目）

査定者がこの9項目の情報を渡せば、Kintone入力から本査定完了メッセージ生成まで完全自動化できます。

1. **レコード番号**（6桁）
2. **3utools画像**（ファイルパス）
3. **付属品の有無**（箱、充電ケーブル、説明書・シムピン）
4. **外装の状態**（表面・背面・側面それぞれ）
5. **不良の内容**（症状対応表A列から選択: BTサービス、OC曇り薄い、光抜け等）
6. **バッテリー最大容量**（実測値）
7. **査定写真**（4枚程度の画像ファイルパス）
8. **出品媒体**（バックマーケット等）
9. **BMコンディション**（A/B/Cグレード）

### 自動処理される項目

以下は査定者から受け取る必要がなく、自動処理されます：

- **充電アダプタ・イヤホンの有無**: 機種から自動判定（iPhone 11 Pro以降は「元々なし」）
- **SIMロック状態**: ほぼ「シムフリー」固定（動作確認後はフリーになる）
- **不良の「不具合など」「修理」選択**: 症状対応表に基づき、顧客にとって最も有利な買取金額になる選択肢を自動判定
- **想定販売価格**: Google Sheets MCP経由で自動取得（機種・容量・BMコンディションから）
- **ランク**: 外装の最悪状態から自動算出
- **本査定完了メッセージ**: 定型文を使って自動生成・修正

## 処理フロー

### 1. Kintoneでレコードを開く

```
URL: https://japanconsulting.cybozu.com/k/11/show#record={レコード番号}
```

### 2. 編集モードにする

「レコードを編集する」ボタンをクリック

### 3. 付属品を設定

#### 付属品の選択肢

- 新品あり
- 中古品あり
- なし
- 元々なし
- 不明

#### 機種別ルール

| 機種 | 操作 |
|------|------|
| iPhone 11以前 | 「全なし」一括選択でOK |
| iPhone 11 Pro以降 | 「全なし」→ **充電アダプタ**と**イヤホン**を「元々なし」に変更 |

**理由**: iPhone 11 Pro以降は元々充電アダプタとイヤホンが同梱されていない

#### iPhone世代判定

- **iPhone 11以前（全なしでOK）**: iPhone 11, XS, XS Max, XR, X, 8, 8 Plus, 7, 7 Plus, SE（第1世代）, 6s以前
- **iPhone 11 Pro以降（元々なし対応）**: iPhone 11 Pro, 11 Pro Max, 12シリーズ以降すべて, SE（第2世代以降）

### 4. 3utools画像をアップロード

- 「3utools画像」フィールドの「参照」ボタンをクリック
- 画像ファイルを選択してアップロード

### 5. 保存

「保存」ボタンをクリック

### 6. ボットによる自動入力を待つ

保存後、Kintoneのボットが3utools画像から以下を自動入力する：

- モデル番号
- キャリア
- ネットワーク利用制限
- IMEI
- バッテリー最大容量
- 3utools読込: 完了

**重要**: ボットの入力結果はページをリロード（更新）しないと反映されない。

#### 確認手順

1. 保存後、約6秒待つ
2. **ページをリロード**する（ブラウザの更新またはF5）
3. 以下のフィールドがすべて入力されているか確認：
   - モデル番号（空欄でないこと）
   - キャリア（空欄でないこと）
   - ネットワーク利用制限（空欄でないこと）
   - IMEI（空欄でないこと）
4. まだ空欄の場合、さらに10秒待ってから再度ページをリロードして確認
5. **すべてのフィールドが入力されるまで、リロード→確認を繰り返す**
6. すべて入力されていることを確認してから、次のステップ7に進む

**注意**: ボットの自動入力が完了する前に編集モードに入ると、保存時に「編集中に、ほかのユーザーがレコードを更新しました」エラーが発生し、すべての編集内容が失われる。必ずボットの完了を確認してから編集を開始すること。

### 7. 再度編集モードにする

ボットによる自動入力が完了し、ページリロードで結果を確認したら、再度「レコードを編集する」ボタンをクリック

- **側面の状態**: ドロップダウンから選択
- **ランク**: 自動入力される（最も悪い状態に基づく）


##### ステップ2: 症状対応表で該当する「不具合など」「修理」を確認

査定者が選んだ不良について、症状対応表のB列（不具合など）とC列（修理）を確認します。

**パターンA: 「不具合など」のみ存在する不良（修理フィールドが空欄）**
- 該当する「不具合など」チェックボックスを自動選択
- 例: 「リンゴループ」→「不具合など」の「正常に起動しない」を選択

**パターンB: 「不具合など」と「修理」両方存在する不良**
- 顧客にとって最も有利な買取金額になる選択肢を自動判定
- 処理手順:
  1. まず「不具合など」にチェック → 買取金額を記録（金額A）
  2. 次に「不具合など」のチェックを外して「修理」にチェック → 買取金額を記録（金額B）
  3. 金額Aと金額Bを比較
  4. **金額が高い方を選択**

- 例: 「光抜け」の場合
  - パターンA：「不具合など」の「表示やタッチ操作の劣化」→ 金額A
  - パターンB：「修理」の「画面修理」→ 金額B
  - 比較して高い方を採用

##### ステップ3: 特記事項の自動追加

以下の不良が選択された場合、Kintoneの「特記事項」フィールドにテキストを自動追加する：

**品質問題なし（質○）パターン**:
不良に「(質○)」が付いている場合、非純正パーツだが使用上の品質に問題がないことを意味する。

| 不良 | 特記事項に追加するテキスト |
|------|--------------------------|
| FP変色(質○) | FP問題ないので交渉アリ |
| FP非純正(質○) | FP問題ないので交渉アリ |
| BT非純正(質○) | BT問題ないので交渉アリ |
| OC非純正(質○) | OC問題ないので交渉アリ |

**修理可能性パターン**:
以下の不良は修理で改善できる可能性があるため、修理係への引き渡しを記録する。

| 不良 | 特記事項に追加するテキスト |
|------|--------------------------|
| Face ID反応悪い | 修理できる可能性があるので修理係に渡す |
| SIMトレー反応なし | 修理できる可能性があるので修理係に渡す |

- 複数該当する場合は、読点（、）で区切って追加する
  - 例: 「FP問題ないので交渉アリ、BT問題ないので交渉アリ」
- Kintoneへの不具合など/修理のチェックは、(質○)なしの通常版と同じマッピングを使用する

##### ステップ4: 最終確認

- 選択した「不具合など」「修理」チェックボックスを確認
- **重要**: 余計な不具合にチェックを入れないよう確認（余計な減額を防ぐため）
- 特記事項の自動追加対象の不良がある場合、テキストが追加されていることを確認

### 9. SIMロックとバッテリー最大容量を入力

- **シムロック**: ドロップダウンから実測値を選択（通常「シムフリー」）
- **バッテリー最大容量**: 実測値を手動入力（3utoolsの値と異なる場合があるため）
  - 例: 67%

**注意**: 3utools画像から自動入力された値と実測値が異なることがあるため、必ず実測値で上書きする

### 10. 査定写真をアップロード

- 「査定写真（これ使って）」フィールドの「参照」ボタンをクリック
- 端末の傷や不良箇所を撮影した画像を選択してアップロード
- 通常4枚程度の画像をアップロード

### 11. 出品媒体・販売経路・想定販売価格を設定

#### 出品媒体
- 該当する出品媒体にチェック（例: バックマーケット）

#### BMコンディション
- ドロップダウンから選択（例: Cグレード）

#### 販売経路と想定販売経路
- 査定時は「**想定販売経路**」を設定（この端末をどこで売る予定かを決定）
- 実際に販売した際に「**販売経路**」を設定
- 査定時は両方とも同じ値を設定（例: バックマーケット）

#### 想定販売価格
- Google Sheetsから該当機種・容量・BMコンディションの平均売価を取得して入力
- または手動で入力

### 12. 本査定完了メッセージを生成・修正

#### メッセージ生成
1. 「メッセージ生成」ドロップダウンから「本査定完了メッセージ」を選択
2. 自動生成されたメッセージを確認

#### メッセージ１（条件価格）の修正
- **価格の修正**: 範囲表記（「6300円〜6300円」）を確定金額（「6300円」）に変更
  - 高額買取価格: 「◯◯円〜◯◯円」→「◯◯円」
  - 特急買取価格: 「◯◯円〜◯◯円」→「◯◯円」
- **改行の調整**: 改行ルールに従って適切に改行を入れる（詳細は「メッセージ生成の改行ルール」参照）
- **「画像のように・・・」の置き換え**: 外装状態と不良内容の定型文に置き換える
  - 外装: 一番悪い箇所（表面・背面・側面のうち）のみを説明
    - 例: 「画像のように側面に複数の傷が確認されましたので、【目立つ傷あり】の評価となりました。」
  - 不良: 該当する定型文を追加
    - 例: 「また、バッテリーの劣化が確認されましたため減額が入っております。」
  - 外装文と不良文の間に改行を入れる

#### 定型文の選択ルール

**外装**: Kintoneの「ランク」フィールドに基づく
- **美品**: 全箇所の定型文
  - 「画像のように、表面・背面・側面に小さなキズが確認されましたので、【美品】の評価となりました。」
- **使用感あり**: 全箇所の定型文
  - 「画像のように、表面・背面・側面に複数の傷が確認されましたので、【使用感あり】の評価となりました。」
- **目立つ傷あり**: 一番悪い箇所のみ特定して記載
  - 例: 「画像のように側面に複数の傷が確認されましたので、【目立つ傷あり】の評価となりました。」
- **外装ジャンク**: 全箇所の定型文
  - 「画像のように、表面・背面・側面に割れ・歪み・欠けが確認されましたので、【外装ジャンク】の評価となりました。」

**不良**: Kintoneの「不具合など」チェックボックスに基づく
- 各不良項目に対応する定型文を使用（詳細は「査定結果ご連絡時の定型文」参照）
- **複数の不良がある場合**: 1文にまとめて記載する（繰り返しを避け、簡潔にする）
  - 症状を読点（、）で列挙し、最後に「減額が入っております。」で締める
  - 例: バッテリー劣化＋アウトカメラ曇り＋アウトカメラガラス割れの場合
    ```
    また、バッテリーの劣化、アウトカメラに曇り、アウトカメラのガラスに割れがございましたため、減額が入っております。
    ```
  - 例: バッテリー劣化＋画面非純正の場合
    ```
    また、バッテリーの劣化、非純正の画面が確認されましたため、減額が入っております。
    ```
  - 各症状の表現は定型文から抜き出し、自然な日本語になるよう調整する
- **不良が1つの場合**: そのまま1文で記載
  - 例: 「また、バッテリーの劣化が確認されましたため減額が入っております。」

**改行ルール**:
- 外装の説明文の後に改行を入れる
- 不良の説明文（1文にまとめたもの）の後に改行を入れる

**外装のみの場合**: 不良がない場合は外装の定型文のみ使用

### 13. 保存

「保存」ボタンをクリックして全ての変更を保存

## 査定フロー全体像

査定は大きく4つのフェーズに分かれる：

1. **動作確認** - 物理的な端末チェック（スキル化対象外）
2. **データ整理** - 本スキルで対応 ✅
3. **連絡** - 顧客への連絡（今後追加予定）
4. **結果連絡** - 査定結果の送信（今後追加予定）

## Google Sheets API設定

### セットアップ

## Slackからのデータ取得

査定者がSlackの「査定写真がお」チャンネルにレコード番号と共に査定データを投稿している場合、以下の手順でデータを取得する。

### Slackデータの構造

査定者は以下の形式でデータを投稿する：

```
レコード番号
外装（頭文字3つ） 不良内容 SIMロック BT容量% 出品媒体 BMコンディション
[3utools画像を添付]
[査定写真を添付]
```

例：
```
339234
ち、し、め BT劣化 SIMフリー 77% バックマーケット C
```

### 取得手順

1. Slackの「査定写真がお」チャンネルでレコード番号を検索
2. テキストデータから以下を抽出：
   - 外装の状態（頭文字: ま/ち/し/め/わ → 表面/背面/側面の順）
   - 不良の内容（症状対応表A列の表記）
   - SIMロック状態
   - バッテリー最大容量（実測値）
   - 出品媒体
   - BMコンディション（A/B/Cグレード）
3. 添付画像をダウンロード：
   - 3utools画像（スクリーンショット）
   - 査定写真（複数枚）

### 注意: 付属品情報はSlackに含まれない

**付属品の有無（箱、充電ケーブル、説明書・シムピン）はSlackのテキストに含まれていない。**
Slackからデータを取得した場合でも、付属品については別途ユーザーに確認する必要がある。

5. 注意事項: 空行を入れて分離
6. 返送案内: 空行を入れて分離

## スピード化案（検討中）

### 概要
査定者がサクッと入力するだけで自動処理を実行する仕組み。

### Slack連携案
```
#査定入力 チャンネル

査定者の投稿:
338174
箱なし ケーブルなし 説明書なし
[3utools画像を添付]
```

### 自動化される部分
1. Kintoneでレコードを開く
2. 編集モードにする
3. 機種を確認して付属品ルールを自動適用
4. 3utools画像をアップロード
5. 保存
6. ボットが端末情報を自動入力

### 査定者が入力するもの（最小限）
1. レコード番号（6桁）
2. 3utools画像
3. 付属品の有無（箱、ケーブル、説明書・シムピンのみ）
   - ※アダプタ・イヤホンは機種から自動判定
4. 外装の状態（表面・背面・側面）
5. 不良の内容

### 外装状態の頭文字入力案

外装状態の選択肢は頭文字がすべて異なるため、頭文字だけで認識可能：

| 頭文字 | 状態 |
|--------|------|
| ま | まったく劣化がない |
| ち | ちいさな傷などがある |
| し | 使用感がある |
| め | 目立つ傷などがある |
| わ | 割れやゆがみがある |

**入力例**: 「表面 ち、背面 ま、側面 め」または「ち ま め」
→ 表面:小さな傷、背面:劣化なし、側面:目立つ傷

### 必要な準備
- Slack MCP連携の設定
- 特定チャンネルの監視
- 画像ダウンロード処理
